# Управление секретами

## Правила

Секреты не коммитятся и не копируются в Markdown, issue, PR, чат, логи, дампы или screenshots. Локальная `.env` и production `backend/.env.production` остаются вне Git. Документация использует только имена переменных и placeholders (`<SECRET>`, `<BOT_TOKEN>`, `<DATABASE_URL>`).

К секретам относятся Bot 1/Bot 2 tokens, JWT/signing keys, PostgreSQL/Redis credentials, admin bootstrap credentials, object/backup storage tokens, Telegram/channel credentials, API keys мониторинга и SSH private keys.

## Жизненный цикл

1. Создавать криптографически стойким способом в доверенной среде.
2. Выдавать минимальному числу сервисов и людей.
3. Передавать через утвержденный secret manager или защищенный канал.
4. Регулярно и после инцидента ротировать с проверкой зависимостей.
5. Отзывать старую версию, проверять логи и документировать событие без значения секрета.
6. Хранить break-glass доступ отдельно и тестировать его.

Перед каждым релизом запускаются secret scanning и проверка `git diff`; на production проверяются только наличие и корректная загрузка переменных без печати значений. Случайно раскрытый секрет считается скомпрометированным и немедленно ротируется.
