# Модель угроз

**Назначение:** рабочая STRIDE-подобная модель для владельца продукта и технической команды. **Срез:** `325e3af`, 13 июля 2026 года.

## Активы и границы доверия

Активы: учетные записи, Telegram-идентификаторы, контакты и адреса доставки, заказы, платежные подтверждения, остатки, возвраты, вложения, секреты ботов/JWT/БД, журналы аудита и резервные копии. Границы проходят между Telegram-клиентом и backend, публичным интернетом и reverse proxy, приложением и PostgreSQL/Redis, seller/admin и магазином, production-хостом и удаленным backup-хранилищем.

| Угроза | Реализованное снижение риска | Остаточный риск / действие |
| --- | --- | --- |
| Подмена Telegram-пользователя | Server-side HMAC и freshness `initData` | Проверить ротацию токенов и негативные тесты в релизе |
| Повтор запроса / устаревший auth | Ограничение возраста и будущего времени | Настроить разумный production TTL, мониторить отклонения часов |
| Межмагазинный доступ | RBAC и seller scope | Провести отдельный IDOR/BOLA аудит всех object endpoints |
| Перепродажа остатка | Транзакция, row lock, атомарное списание | Нагрузочный тест конкурентного checkout |
| Несанкционированная смена заказа/платежа | Роли, guards, `AuditLog` | Линейность статусов заказа программно не гарантирована |
| Злоупотребление загрузкой | Типы и лимит 20 MiB для return evidence | Нужны malware scan, quarantine и строгая раздача файлов |
| Утечка секретов | env вне Git, placeholders в документах | Формализовать vault/rotation/break-glass |
| Спам и истощение ресурсов | Redis rate limiting и fallback | Fallback не является распределенным; нужны метрики и edge limits |
| Потеря/шифрование данных | Локальные и удаленные backup, restore verify | Проверить RPO/RTO и независимость удаленного хранилища |
| Утечка через логи | Запрет raw `initData`, санитизация | Регулярно сканировать логи и telemetry payloads |
| Компрометация Bot 1/Bot 2 | Разделение обязанностей | Отдельные токены, минимальные права, план немедленной ротации |
| Supply-chain атака | Lockfiles и CI-проверки | Включить SCA, SBOM, secret scanning и подпись релиза |

## Критические сценарии проверки

Перед продажами проверить IDOR, privilege escalation, Telegram replay, CSRF/CORS политику, injection, небезопасную раздачу uploads, SSRF через URL, webhook spoofing, brute force, dependency CVE, backup disclosure и восстановление после полной потери БД. Результаты и принятый риск должны иметь владельца и дату.
