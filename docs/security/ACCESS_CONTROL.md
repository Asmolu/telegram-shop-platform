# Контроль доступа

## Прикладной уровень

Роли и разрешения описаны в [Roles and Permissions](../product/ROLES_AND_PERMISSIONS.md). Покупатель действует в собственном контексте; seller ограничен своим магазином; admin выполняет платформенные операции. Bot 1 обслуживает покупателей, Bot 2 — seller/admin/auth. Критические seller/admin действия должны создавать `AuditLog`.

Каждый endpoint обязан проверять не только роль, но и принадлежность объекта магазину/пользователю. Клиентское скрытие кнопки не является контролем доступа. Service отвечает за бизнес-правила, repository — за scoped query; router не должен содержать авторизационную бизнес-логику.

## Инфраструктурный уровень

SSH, production host, CI, домены, Telegram bots, backup и БД используют персональные учетные записи, MFA где доступно, least privilege и журналирование. Общие бессрочные credentials недопустимы. Доступ увольняемого/сменившего роль сотрудника отзывается по чек-листу.

Не реже ежеквартально владелец системы сверяет список доступов, неиспользуемые ключи, администраторов Telegram и права backup-хранилища. Emergency access выдается ограниченно по времени и разбирается постфактум.
