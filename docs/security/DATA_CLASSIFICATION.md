# Классификация данных

**Статус:** внутренняя политика-проект; юридическое утверждение требуется до запуска.

| Класс | Примеры | Обращение |
| --- | --- | --- |
| Публичные | каталог, цены, публичные отзывы, домены | Можно публиковать после модерации и проверки прав |
| Внутренние | архитектура, runbooks, метрики, audit metadata | Доступ по рабочей необходимости; не публиковать целиком |
| Персональные | Telegram user id/username, имя, телефон, адрес, размеры, история заказов, обращения | Минимизация, цель и срок хранения, ограниченный доступ, обработка запросов субъекта |
| Чувствительные операционные | платежные доказательства, вложения возврата, IP/UA и security logs | Усиленный доступ, контролируемая выдача, короткие обоснованные сроки |
| Секретные | bot tokens, JWT secrets, DB/Redis credentials, backup credentials, private keys | Только secret store/env на целевой среде; никогда в Git, документации, логах и тикетах |

Номер банковской карты не должен собираться приложением для текущего ручного сценария СБП. Telegram `initData` является аутентификационным материалом и не должен сохраняться или логироваться. В БД для файлов хранятся только ссылки/пути, однако сами файлы наследуют класс содержащихся в них данных.

Владелец продукта совместно с юристом определяет правовые основания, сроки хранения и процедуру удаления. Техническая команда реализует утвержденную матрицу, а не назначает ее самостоятельно.
